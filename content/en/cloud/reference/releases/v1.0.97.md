## Hotfix - production sign-in restored on cloud.layer5.io

Fixes a regression where clicking **Sign in with GitHub** / **Sign in with
Google** on `cloud.layer5.io/login` redirected users to the upstream OAuth
provider with a `redirect_uri` of
`https://cloud.layer5.io/kratos/self-service/methods/oidc/callback/{provider}`,
which GitHub / Google rejected as edirect_uri is not associated with this
application\ after the production OAuth App callback URL was updated to the W2
native path `/oidc/callback/{provider}`.

### What changed

- `OrgAuthData` exposes a new `UseCloudNativeOidc` boolean that is true for both
  registered custom-domain hosts AND for the canonical Provider Org host
  (`SERVER_BASE_URL`'s host). The React login / registration pages now gate the
  W2 cloud-native redirect on this field, so canonical-host sign-in no longer
  falls through to the legacy Kratos OIDC flow.
- A new `resolveOIDCOrgID(req)` helper in the server handlers prefers the
  middleware-resolved org and falls back to `ProviderOrgID` when the request
  lands on the canonical host. Both `OIDCStart` and `OIDCCallback` consume the
  helper so the state binding's `orgId` is symmetric across the round trip on
  the canonical host.

### Rollout notes

The Layer5 production GitHub OAuth App's Authorization callback URL is already
`https://cloud.layer5.io/oidc/callback/github` and this deploy restores
canonical-host sign-in without further OAuth App changes.

The staging GitHub OAuth App (1065875) must mirror this with
`https://staging-cloud.layer5.io/oidc/callback/github` for staging sign-in to
work after this deploys to staging. If that has not yet been updated, staging
will exhibit the same symptom until it is.
